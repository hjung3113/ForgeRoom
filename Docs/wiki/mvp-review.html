ForgeRoom Forge Phase 1 MVP

ForgeRoom MVP는 로컬 단일 머신에서 실행되는 중앙 멀티에이전트 오케스트레이터다. 사용자가 Discord slash command 또는 GitHub Issue label로 작업을 요청하면, ForgeRoom이 대상 프로젝트의 context를 선별하고, task 전용 branch와 worktree를 만들고, workflow DSL에 따라 여러 CLI agent를 순차 실행한 뒤, test/lint/typecheck를 통과한 변경을 PR로 만든다.

Runtime: Node.js + TypeScript Storage: SQLite + Drizzle Agent runtime: AgentRuntimeProvider MVP provider: OpenClawProvider Prompt IO: file-based Context: ForgeMap

기준 문서: overview.md, architecture.md, phase-1-mvp.md

이 문서의 성격

이 HTML은 canonical spec이 아니라 리뷰용 wiki다. 설계 변경은 ADR과 원본 문서에 반영되어야 한다. 여기서는 현재 문서들이 말하는 MVP를 한 번에 검토할 수 있도록 흐름, 모듈, 결정, 리스크를 재배열했다.

2. 범위와 비범위

MVP가 제공하는 것

영역MVP 내용리뷰 포인트
Task intake Discord slash command와 GitHub.com Issue label polling을 TaskSource로 사용한다. 내부 기업 환경에서 Discord/GitHub.com이 실제 MVP 입력 경로로 충분한가?
Project context ForgeMap이 대상 프로젝트의 canonical context를 관리하고, task 시작 시 필요한 subset만 worktree에 staging한다. ForgeMap의 최소 산출물과 stale/dirty 정책이 운영 가능한가?
Execution Workflow DSL이 plan, review, refine, slice 구현, code review loop를 순차 실행한다. default workflow가 너무 무겁거나 너무 암묵적이지 않은가?
Agent runtime AgentRunner는 provider-neutral 계약만 알고, MVP 구현체 OpenClawProvider가 실제 CLI runtime 호출을 맡는다. OpenClawProvider 하나로 MVP 검증이 가능한가?
Quality CheckRunner가 execute step 직후 project commands를 실행하고, 실패 시 1회 자동 수정 요청 후 재검증한다. check fix budget 1회가 적절한가?
Output Check 통과 후 branch를 push하고 PR을 생성한다. 사람은 GitHub UI에서 merge한다. PR body와 status comment에 필요한 정보가 충분한가?

  • Discord 승인 게이트

제품 확장

  • 동적 workflow 변형
  • Desktop GUI
  • Tailscale, inbound remote access, multi-machine queue

3. 사용자와 입력 경로

Human maintainer

작업을 요청하고, 중간 산출물을 확인하고, 필요하면 `/feedback`으로 지시를 보강한다. 최종 merge는 사람이 GitHub에서 수행한다. ForgeRoom은 merge 자동화가 아니라 PR 생성까지를 MVP 자동화 경계로 둔다.

Discord command user

`/run`, `/pause`, `/resume`, `/cancel`, `/status`, `/ask`, `/feedback`으로 task lifecycle에 개입한다. Gateway는 명령을 수신하고 TaskRequest를 만들며, 진행 보고는 Reporter가 맡는다.

GitHub issue author

등록된 repo의 issue에 trigger label을 붙이면 GitHubGateway가 polling으로 감지해 task를 생성한다. status는 하나의 pinned status comment를 갱신하는 방식으로 누적된다.

Agent runtime

Claude Code, Codex 같은 CLI agent는 ForgeRoom의 직접 관심사가 아니다. ForgeRoom은 OpenClawProvider를 통해 provider-neutral request를 전달하고 output file을 검증한다.

핵심 설계 원칙

  • 중앙 orchestrator는 로컬 단일 프로세스다.
  • 대상 프로젝트에는 최소한의 파일만 둔다. ForgeRoom 운영 artifact는 task worktree의 `.forgeroom/` 아래에 둔다.
  • 1 Task는 보통 1 Issue를 처리하고, 1 Branch / 1 Worktree / 1 PR을 갖는다.
  • Agent들은 같은 worktree를 turn-taking 방식으로 사용한다.
  • Prompt와 output은 파일 기반으로 주고받는다.
  • Agent runtime 실행과 project verification은 분리한다. CheckRunner는 ForgeRoom이 직접 실행한다.

Report + PR
Reporter가 상태를 갱신하고, 마지막 check 통과 후 GitHubGateway가 PR을 만든다.

Step 내부 흐름

  1. Resolved Step 생성: workflow step + intent + prompt template + vars + input_refs를 합친다.
  2. 변수 보간: `${task.*}`, `${step.output_path}`, `${task.final_slices}` 등을 해석한다.
  3. 사용자 피드백 통합: pending feedback event가 있으면 Conductor가 `feedback.md`를 갱신한다.
  4. Prompt 보강: Conductor가 ForgeMap subset, summary, feedback, 직전 결과를 반영한다.
  5. Prompt 저장: `.forgeroom/prompts/NN_step.md`에 쓴다.
  6. Agent 실행: AgentRunner가 provider에 prompt/output path와 cwd를 전달한다.
  7. Output 검증: 파일 존재, 최소 크기, 명백한 거부 응답, selector contract를 확인한다.
  8. Execute step이면 CheckRunner를 실행한다.
  9. Diff 저장, step 완료, Conductor.update, Reporter.notify를 수행한다.

6. Workflow DSL

Workflow는 agent에게 바로 명령을 쓰는 파일이 아니라, step 순서와 제어 흐름을 선언하는 library다. 실행 구성은 `configs/intents.yaml`에 둔 Intent Catalog가 소유하고, step은 intent와 prompt template을 참조한다.

개념역할MVP 제약
Workflow step 순서, group, review_loop, effects metadata를 정의한다. `effects.worktree`, `effects.external.report`, `effects.external.pr` 필수.
Intent Intent Kind, agent, Step Harness를 묶은 실행 preset이다. step이 `kind`, `agent`, `harness`를 직접 지정하지 않는다.
Prompt Template 해당 step에서 실제로 무엇을 시킬지 렌더링하는 지시 양식이다. bundled template root 아래 상대 경로만 허용. inline prompt 금지.
foreach slice list를 순회하며 내부 steps를 실행한다. MVP slice 구현은 `${task.final_slices}`만 사용한다.
review_loop review가 pass할 때까지 review/refine 쌍을 반복한다. `until`은 `${review_id.passed}` 형식만 허용한다.

Output contract

Workflow Control Model

stateDiagram-v2

    ResolveStep --> RenderPrompt
    RenderPrompt --> ConductorRefine

    AgentRun --> OutputContract

    Failed --> [*]

Slice extraction

`implementation_plan.md`와 `refine_plan.md`는 마지막에 `## Slices` 섹션을 내야 한다. PipelineEngine은 top-level `- ` bullet만 slice 문자열로 인정한다. 0개면 output contract 실패다.

Review result

`kind: review` output의 첫 non-empty line은 정확히 `Review Result: pass` 또는 `Review Result: fail`이어야 한다. review_loop 종료 조건은 이 헤더만 신뢰한다.

Default workflow의 의미

기본 제공 workflow는 온보딩용 sane default다. 실제 운영은 project의 `allowed_workflows` 안에서 custom workflow를 선택하는 모델이다. Custom workflow도 Intent/Agent/Harness registry를 통과해야 하므로, 임의 agent 호출을 workflow 파일에서 바로 열어두지 않는다.

ContextSelector

task title, description, workflow, changed refs, symbol matches, dependency hints를 기준으로 subset을 고른다. MVP는 direct match와 명시적 1-hop dependency를 기본으로 한다.

Selection log

포함된 문서마다 reason과 signal을 남겨야 한다. 예: `matched_symbol: DiscordGateway`, `1hop_depends_on: DiscordGateway -> ReporterSink`.

Stale check

`indexed_commit`과 current HEAD가 같고 dirty change가 없으면 기존 ForgeMap을 사용한다. HEAD 변경, dirty baseline, pending rebuild는 별도 정책을 탄다.

Conductor relation

Conductor는 staged context를 읽고 prompt를 보강하지만 canonical ForgeMap을 직접 수정하지 않는다. task 종료 후 refresh 계획은 ForgeMapRefresher가 맡는다.

Modification workflow에서 ForgeMap이 pending rebuild 상태면 기본 중단한다. Maintainer approval이 있으면 warning artifact를 남기고 진행할 수 있지만, 이는 리뷰해야 할 운영 정책이다.

8. Agent Runtime 경계

ForgeRoom은 agent process 실행 세부사항을 직접 소유하지 않는다. AgentRunner는 provider-neutral request를 만들고, AgentRuntimeProvider가 OpenClaw 같은 runtime gateway에 연결한다.

경계책임책임이 아닌 것
AgentRunner prompt/output/log path 할당, provider 호출, output 검증, retry/resume 판단. provider-specific auth, runtime session internals, test/lint 실행.
AgentRuntimeProvider `run`, `resume`, `health`를 제공하는 adapter interface. ForgeRoom retry semantics 결정, output contract 검증.
OpenClawProvider MVP provider. runtime/model/cwd와 prompt instruction을 OpenClaw IPC/HTTP에 전달. Discord/GitHub command handling, project verification command 실행.
CheckRunner project commands 실행, exit code/stdout/stderr/timeout 기록. Agent runtime 호출 경유 실행.

Retry budget

  • AgentRunner output-producing attempt budget 기본값은 3회다.
  • provider non-zero exit, timeout, output 미작성, output 너무 작음, selector contract 실패가 같은 budget을 쓴다.
  • CheckRunner 자동 수정은 별도 budget이며 MVP에서는 1회다.
  • Provider는 자체 retry 정책을 갖지 않고, AgentRunner가 resume 또는 신규 headless fallback을 결정한다.

9. 품질 게이트와 안전장치

CheckRunner

`projects.yaml`의 `commands`를 정의된 순서대로 worktree cwd에서 실행한다. 하나라도 실패하면 실패 로그 마지막 200줄을 prompt로 만들어 마지막 코드 작성 agent에게 1회 수정 요청한다.

ApprovalGate

위험 명령, 위험 경로, main 직접 수정, secret 접근, migration/reset, `curl | sh` 같은 작업을 거부한다. MVP는 승인 흐름이 아니라 거부만 제공한다.

Conductor scope defense

Conductor는 코드 작성자가 아니다. 호출 전후 git status를 비교하고, 허용 파일 외 변경이 있으면 revert하고 scope violation log를 남긴다.

Reporter delivery

외부 알림은 outbox delivery로 처리한다. Discord/GitHub status update 실패는 task 실행을 실패시키지 않고 재발송 대상으로 남긴다.

Check status model

Status의미
not_run`kind: execute`가 아니거나 아직 CheckRunner가 실행되지 않은 상태.
passed첫 CheckRunner 실행에서 모든 command 통과.
fixed첫 실행은 실패했지만 자동 수정 1회 후 모든 command 통과.
failed자동 수정 없이 실패했거나 자동 수정 후에도 command 실패.

10. 저장소와 복구

TaskStore는 SQLite 기반의 영속 저장소이자 project별 FIFO queue 역할을 한다. DB는 task history와 external delivery 상태를 기록하고, prompt/output 본문은 파일을 primary source로 둔다.

Table역할핵심 포인트
taskstask lifecycle, project lock, external ref.project별 active task는 최대 1개.
stepsworkflow step 실행 이력.step retry는 같은 row의 attempt 증가. 새 row 생성 아님.
checksCheckRunner command 결과.append-only로 exit code, log path, duration을 남긴다.

      string task_id
      string type
      string payload
      datetime created_at
    }
    EVENT_DELIVERIES {
      string id
      string event_id
      string sink
      int delivery_attempts
      datetime next_delivery_at
      datetime delivered_at
    }
    CONDUCTOR_STATE {
      string task_id
      string summary_path
      datetime updated_at
    }

Recovery

  • Orchestrator 재시작 시 `status IN ('running','paused')` task를 조회한다.
  • 마지막 step row와 control step child rows를 보고 다음 실행 지점을 복원한다.
  • worktree의 `.forgeroom/` 디렉토리를 검증하고 없으면 bootstrap을 재실행한다.
  • canceled task는 자동 resume 대상이 아니며, worktree/branch/PR은 분석 또는 수동 이어받기를 위해 보존한다.

WorktreeManager task별 branch/worktree 생성, `.forgeroom/` bootstrap, idempotent reuse. main/default branch 직접 worktree와 project path 내부 worktree 금지. worktree-manager.md

Conductor summary 유지, prompt 보강, feedback 통합, `/ask` 응답. 코드 작성, commit, PR 생성 금지. conductor.md
CheckRunner execute step 직후 project commands 실행, 실패 시 1회 자동 수정. Agent runtime 호출이 아니라 ForgeRoom 직접 실행 품질 게이트. check-runner.md
Reporter ReporterEvent 생성, Discord/GitHub delivery, status surface 멱등 갱신. status delivery 실패는 task 실패가 아님. PR 생성 실패는 workflow failure 가능. reporter.md
DiscordGateway slash command 수신, allowlist 검증, TaskRequest 생성. 진행 보고는 Reporter 책임. Gateway는 라우팅만. discord-gateway.md
GitHubGateway GitHub.com Issue label polling, PR 생성, status comment marker 복구. GitHub Enterprise는 Phase 2. MVP는 GitHub.com API 검증. github-gateway.md
ApprovalGate 위험 명령·경로·workflow를 시스템 레벨에서 거부. MVP는 승인 없음, 거부만. approval-gate.md

13. 완료 정의

  1. 프로젝트 1개 등록 후 Discord `/run`으로 풀 파이프라인 자동 실행.
  2. `full`, `quick`, `hotfix` workflow 호출 정상 작동.
  3. 단계별 Discord 알림 도착.
  4. `/ask <task>`로 task context 질문 응답.
  5. `/feedback <task>`로 사용자 피드백을 다음 step prompt에 반영.
  6. ForgeMap 최초 생성 후 task 시작 시 관련 context가 `.forgeroom/context/`에 staging.
  7. PR 자동 생성, GitHub UI에서 merge 가능.
  8. Orchestrator 재시작 후 진행 중 task 이력 복구와 다음 step resume.
  9. 위험 명령 거부 검증.
  10. Step output 파일 미작성 시 자동 재시도.
  11. Conductor scope 위반 시 git revert.
  12. `/cancel <task>`로 active task를 canceled로 전환하고 같은 project queue 진행 가능.

15. 리뷰 체크리스트

Scope

  • Discord/GitHub.com을 MVP TaskSource로 두는 것이 실제 검증 환경에 맞는가?
  • OpenCodeProvider와 Local CLI를 Phase 2로 미루는 것이 맞는가?
  • ForgeMap을 MVP 핵심 기능으로 넣는 것이 MVP 비용을 과도하게 키우지 않는가?
  • Desktop/Tailscale/multi-machine 제외가 충분히 명확한가?

Execution

  • 1 task = 1 branch/worktree/PR 불변식이 모든 flow에 맞는가?
  • pause/resume/cancel semantics가 운영자가 기대하는 방식인가?
  • review_loop의 max_iterations 실패 처리가 충분히 진단 가능한가?
  • output-producing attempt budget과 check fix budget 분리가 이해하기 쉬운가?

Context

  • ForgeMap 파일 구성은 실제 target project를 설명하기에 충분한가?
  • ContextSelector의 1-hop 제한이 context 누락을 너무 자주 만들지 않는가?
  • dirty baseline과 pending rebuild block이 너무 보수적이거나 느슨하지 않은가?
  • Conductor가 canonical ForgeMap을 수정하지 않는 경계가 충분히 강한가?

Operational Safety

  • ApprovalGate의 MVP 거부 catalog가 실제 위험을 충분히 막는가?
  • Secret 접근 차단과 worktree 외부 write revert가 충분한가?
  • Reporter delivery failure가 task 실패와 분리되는 것이 맞는가?
  • PR 생성 실패를 task failed로 보는 기준이 적절한가?

리뷰 중 표시하면 좋은 항목

  • 범위 변경: MVP에 넣거나 빼야 하는 기능.
  • 경계 불명확: 어떤 모듈 책임인지 애매한 부분.
  • 운영 리스크: 실제 사용 중 막힐 가능성이 높은 흐름.
  • ADR 필요: 기존 결정과 충돌하거나 새 결정이 필요한 변경.
  • Phase 이동: Phase 1/2/3/4 배치가 어색한 항목.
리뷰 결과가 설계 변경으로 이어지면 `Docs/rules/doc-rules.md`의 변경 워크플로우를 따른다. 의미 있는 변경은 ADR proposed에서 시작해 사용자 승인 후 decided로 전환하고, 영향을 받은 문서를 같은 변경 단위로 갱신한다.
